## 1. Rendering/game framework comparison for LLM codegen

| Framework | LLM code burden for a working game | Physics | Mobile/touch | Approx. size | Training-data prevalence | Determinism | Failure modes when code is slightly wrong |
|---|---|---|---|---|---|---|---|
| **Phaser 3** | Lowest: scenes, loader, input, audio, tweens, particles, camera FX built in | Arcade + Matter.js built in | Good; touch/pointer unified, Scale Manager | ~1.2MB min (~500KB common builds) | **Highest** of any game framework (10+ yrs of tutorials/forums) | Weak (variable timestep default; Arcade not deterministic) | Usually degrades gracefully (silent no-op, missing sprite); lifecycle/`this` scoping and asset-key typos are classic LLM errors |
| **Phaser 4** ("Caladan" v4.0 Apr 2026, v4.1 "Salusa") | Same API surface, new render-node WebGL renderer, unified Filter system, SpriteGPULayer (1M sprites/draw) | Same | Same | Similar | Low — training data is Phaser 3; API "mostly kept" so v3-shaped code often works, but filters/FX APIs changed | Weak | LLMs will emit v3 FX/pipeline code that breaks on v4 |
| **PixiJS v8** | High: renderer only — no physics, scenes, input mapping, audio | None (add planck/matter/rapier) | Pointer events good; you build the rest | ~450KB | High | Good (it does nothing per-frame you don't write) | Blank screen or exceptions; but small API = fewer hallucination targets |
| **KAPLAY** (Kaboom fork, community-run since May 2024; v3001 stable line, v4000 in progress) | Very low: component-ES, one-call sprites, built-in physics-lite, scenes, `onKeyDown`, camera shake | Basic arcade-style built in | Decent touch support | Moderate (~few hundred KB) | Medium — Kaboom-era Replit tutorials abound, but v3001/v4000 API drift confuses models | Weak | API drift between kaboom/v3001/v4000 is the #1 LLM failure; smaller community for edge cases |
| **Excalibur.js** | Low-ish: batteries-included TS engine (~300KB) | Built-in arcade + realistic solver | Good | ~300KB | Low (still v0.32, pre-1.0) | Fixed-update option | Pre-1.0 breaking changes; sparse training data → hallucinated APIs |
| **three.js** | High for a *game* (renderer only: no loop conventions, physics, input, UI) | None (rapier/cannon-es) | Touch = DIY | ~170KB gz core | **Very high** — dominates LLM corpora; the proven vibe-coding target (fly.pieter.com, 1,170-entry 2025 Vibe Coding Game Jam was three.js-heavy; Rosebud AI standardized on three.js) | Neutral | Camera/lighting/scale mistakes → black screen; per-frame allocation → GC jank |
| **react-three-fiber v9.5+ / @react-three/rapier v2 / ecctrl** | Medium: declarative JSX scene + ready character controller; React 19 required | Rapier (WASM, has deterministic build flag) | Touch = DIY | three + React + rapier (heavy) | High and growing | Rapier `enhanced-determinism` is cross-platform deterministic | React re-render loops killing frame rate; hooks-in-loop errors |
| **PlayCanvas** (v2.19, MIT) | Medium; engine ~150KB, mobile-optimized, but real value is its cloud Editor — useless for headless LLM gen | Ammo/rapier integration | Excellent, mobile-first | ~150KB engine | Medium-low for code-only use | Neutral | Editor-centric docs mislead code-only generation |
| **Godot 4 web export** | N/A for direct codegen; LLM writes GDScript, platform exports | Full engine | OK on Android; **iOS Safari chronic problems** (infinite-load with threads, memory-kill reloads; must export single-threaded) | 30–100MB+ exports; SharedArrayBuffer/COOP-COEP headaches | High (GDScript well represented) | Poor for web | Export toolchain itself is the failure mode — disqualifying for instant-publish |
| **Raw Canvas2D** | Highest total code, but zero framework to misuse; LLMs one-shot small Canvas2D games reliably | None | DIY | 0KB | Maximal (MDN etc.) | Excellent | Errors are plain JS bugs; perf ceiling ~1–2k sprites |

Key insight from the vibe-coding wave: LLMs succeed when (a) the API is heavily represented in training data, (b) the version is pinned and stable, and (c) errors fail loud. Phaser 3 and three.js win (a); nothing wins (b)+(c) unless *you* own the API layer — which is the argument for the wrapper.

## 2. The opinionated wrapper: fantasy-console precedents

- **PICO-8**: 128×128, 16 colors, 8192 tokens, fixed `_init/_update/_draw` 30/60Hz loop, 6-button input. Constraints (1) collapse decision space so everything ships, (2) make every game automatically consistent/responsive/pleasant, (3) make the platform itself the aesthetic. "Once you know your scope, you're creative *within* it." ([Wikipedia](https://en.wikipedia.org/wiki/PICO-8), [Atomic Object](https://spin.atomicobject.com/pico-8-review/), [Medium](https://medium.com/swlh/creativity-through-limitation-pico-8-fantasy-console-175294e13332)). People already vibe-code PICO-8 games successfully — the tiny API is LLM-friendly ([lgallardo](https://lgallardo.com/2025/11/09/pico8-ai-game-development/)).
- **Bitsy/PuzzleScript**: deliberately shrink the design space; fewer possible games → every game works, and the engine can deeply optimize its one genre ([Pomona notes](https://cs.pomona.edu/classes/cs181g/notes/games-vs-engines.html), [Warren, Tiny Online Game Engines](https://mywebspace.quinnipiac.edu/jbwarren/archive/2022-175/images/inception/WarrenJB-Tiny-Online-Game-Engines.pdf)). Lesson: constraint level is a dial — PuzzleScript is *too* narrow for this platform; PICO-8 is roughly the right altitude.
- **LÖVE / Playdate SDK**: minimal fixed-callback loops (`love.update/draw`) + curated stdlib; proof that "loop owned by platform, logic owned by author" scales to real games.
- **Fortnite Creative/Verse (and Crayta)**: capability-scoped language + curated device APIs means UGC code *cannot* break the platform, crash other players, or escape moderation — the enterprise-grade version of the same idea.

**A "fantasy console for LLMs"** (synthesizing the above + Poki requirements): platform-owned **fixed-timestep loop** (deterministic accumulator, LLM writes `init/update(dt,input)/draw` or entity components); **entity helpers** (spawn/collide/destroy, tags, simple arcade physics); **built-in juice** as one-liners (`shake()`, `burst()`, `tween()`, `flash()`, sound synth presets) so every game feels good without the LLM writing particle math; **unified input** (`input.dir` vector + `input.a/b` buttons mapped to WASD/arrows, auto virtual joystick+buttons on touch, gamepad free); **fixed virtual resolution** (e.g. 960×540 landscape / 540×960 portrait, declared per game) with automatic fit+letterbox; **seeded RNG only** (`rand()` from platform, `Math.random` banned) and no wall-clock access → determinism for replays; **hard budgets** (entity cap, texture cap, update-time watchdog) so slightly-wrong code degrades instead of freezing the tab. The wrapper is also the validation surface: lint/execute generated code headlessly against the API schema before publishing.

## 3. Multiplayer infra for instantly-published UGC games

2026 market reality check: **Hathora shut down May 5, 2026** (team joined Fireworks AI; customers migrated to Nitrado GameFabric) ([Hathora blog](https://blog.hathora.dev/hathora-is-joining-fireworks-ai/), [GamesBeat](https://gamesbeat.com/hathora-acquired-will-exit-game-infrastructure-biz-and-hand-over-customers-to-nitrado/)). **Rivet pivoted** from game servers to "Rivet Actors" for AI/stateful workloads — notably now with first-class docs for [AI and user-generated actors](https://rivet.dev/docs/actors/ai-and-user-generated-actors/) and Rivet Compute (June 2026), so it re-enters the shortlist from the other direction. **PartyKit** was acquired by Cloudflare (2024); the maintained successor is **partyserver** (v0.5.8, June 2026) on Durable Objects ([repo](https://github.com/cloudflare/partykit)). **Colyseus** is independent again, 0.16 (Feb 2026) added StateView + experimental WebTransport, aiming at 1.0 in 2026, with Colyseus Cloud hosting ([blog](https://colyseus.io/blog/colyseus-016-is-here/)). **Playroom Kit** is the highest-level option: no backend at all, client/host-authoritative shared state, MAU pricing (~$10/mo per 10k MAU), strong Discord Activities story ([docs](https://docs.joinplayroom.com/comparison)).

Transports: WebSocket remains the baseline; **WebTransport** finally became cross-browser when Safari 26.4 shipped it, but older iOS requires WebSocket fallback, so WS-first is still correct for 2026 ([websocket.org comparison](https://websocket.org/comparisons/), [Ably](https://ably.com/blog/can-webtransport-replace-websockets)). WebRTC DataChannels only make sense P2P (host-authoritative rooms) — that's Playroom's model; fine for ≤8-player party games, wrong for Agar.io arenas (host cheating, host disconnect, upload bandwidth).

**Untrusted generated code server-side** is the crux. Options: (a) **Cloudflare Dynamic Worker Loader** (open beta April 2026) — spin up V8 isolates for AI-generated code in milliseconds with capability-scoped bindings ([InfoQ](https://www.infoq.com/news/2026/04/cloudflare-dynamic-workers-beta/)); (b) **QuickJS-in-WASM** inside your own process (e.g. [kyushu](https://github.com/peterpeterparker/kyushu)-style wasmtime sandboxing) — slower (interpreted) but hermetic and trivially deterministic/meterable; (c) Rivet's [secure-exec](https://github.com/rivet-dev/secure-exec) / isolate approaches.

**Pragmatic 2026 stack for "LLM writes a shared game-state reducer, platform runs it in a room"**: one **Durable Object per room** (via partyserver) accepting WebSockets with hibernation; the DO loads the game's generated reducer either via Dynamic Worker Loader isolate or QuickJS/WASM embedded in the DO; clients send inputs, reducer folds `(state, inputs, tick) → state`, DO broadcasts snapshots/deltas at 10–20Hz with client-side interpolation. DO hibernation makes idle UGC rooms nearly free — the economic unlock for thousands of mostly-dead generated games ([CF hibernation docs](https://developers.cloudflare.com/durable-objects/examples/websocket-hibernation-server/)). Cloudflare advises "low thousands of sockets per DO," sharding by room — comfortably covers 100-player arenas; the ceiling is reducer CPU per tick, so arena-scale games need spatial-partitioning helpers in the platform SDK, not in LLM code.

## 4. Leaderboards + anti-cheat

- **Client-reported scores are trivially spoofed**; signing requests doesn't help since the key ships in the client. The .io answer is architectural: Agar/Slither/Diep are **server-authoritative** — clients send only intent, server computes score, so score spoofing is impossible by construction ([AccelByte](https://accelbyte.io/blog/server-authoritative-logic-to-prevent-cheating), [crux](https://crux.supercraft.host/blog/server-authoritative-anti-cheat-backend/)). Bots/aimbots (legal-but-inhuman inputs) still survive; handle behaviorally.
- **Single-player leaderboards**: the gold standard is **deterministic input-replay validation** (Open Hexagon): client submits input log + RNG seed; server re-simulates with the same reducer and accepts the computed score ([vittorioromeo](https://vittorioromeo.com/index/blog/oh_secure_leaderboards.html)). This is exactly what a fantasy-console wrapper enables for free: fixed timestep + seeded RNG + no-wall-clock = deterministic sim; the same sandbox infra can run the code server-side. Guard the replayer with CPU/time budgets (replays cost ~sim-time; run at max speed, cap duration).
- **"Accept cheating, design around it"**: Poki-style portals have no accounts and no global leaderboards — cheating is irrelevant there; .io games use ephemeral per-room leaderboards where a cheater costs one round, not a persistent economy. Pragmatic tiering: per-room/daily boards = accept + anomaly heuristics (score-vs-time outliers, z-score caps); persistent/global boards = replay validation; multiplayer = server-authoritative reducer (already the architecture).

## 5. Responsiveness (Poki-informed)

Poki's [quality guidelines](https://sdk.poki.com/poki-quality-guidelines) and [requirements](https://sdk.poki.com/requirements): responsive across screen sizes **and input methods**; force mobile control schemes on tablets; use touch events (not click); show the right control hints per device; initial download **under ~8MB** and playable fast; support **both portrait and landscape** where possible (portrait unlocks mobile ad formats and half of mobile traffic). Technique consensus ([William Malone](https://www.williammalone.com/articles/html5-game-scaling/), [gamedevjs](https://gamedevjs.com/articles/best-practices-of-building-mobile-friendly-html5-games/)): fixed virtual resolution + uniform scale; prefer **fit/letterbox** for correctness, with an "extend" variant (widen world bounds into the letterbox area) for polish — the platform, not the LLM, should implement this. Touch overlays: dynamic (spawn-under-thumb) virtual joystick with dead zone + spring-return, buttons sized as % of screen with generous hit areas, visually obvious active zones ([nipplejs pattern](https://emanueleferonato.com/2023/07/24/add-a-virtual-joystick-to-your-html5-games-in-a-matter-of-minutes-with-nipplejs-example-powered-by-phaser-and-arcade-physics/), [aaronbell](https://www.aaronbell.com/mobile-touch-controls-from-scratch/)). Make orientation a **declared game property** (chosen at design time from genre); render the overlay from the declared input scheme.

## Original stack recommendation (superseded, kept for v2 reference)

**Primary (as researched)**: Custom fantasy-console SDK on PixiJS v8; platform owns loop/scale/input/juice/RNG/audio; LLM writes logic against a small typed API (ship the full API `.d.ts` + examples in the generation prompt). Multiplayer: Cloudflare Durable Objects via partyserver, WebSocket + hibernation, LLM-written deterministic reducer in Dynamic Worker Loader isolates (fallback: QuickJS/WASM in-DO). Same reducer runs client-side for prediction and server-side for authority and replay-validated leaderboards.

**Alternate A (lower build cost — closest to what the founder chose)**: Phaser 3 (pinned) + thin conventions layer + Playroom Kit for ≤8-player rooms. Maximizes LLM one-shot success from training data; defers arena-scale multiplayer and robust anti-cheat. Beware Phaser 4: better engine, wrong training-data era — pin v3 until corpora catch up.

**Alternate B (3D track)**: three.js (plain, not R3F — avoid React re-render pitfalls in generated code) + rapier with `enhanced-determinism`, same wrapper philosophy. Colyseus 0.16+ self-hosted is the fallback if DO CPU limits pinch on 100-player arenas.

## Open questions from research

1. **Isolate economics**: Dynamic Worker Loader is beta — pricing per spawned isolate at UGC scale is unpublished; QuickJS-in-DO may be cheaper but ~10-50x slower per tick. Needs a benchmark spike with a 100-entity arena reducer.
2. **Determinism boundary**: JS float math is deterministic per-engine but a reducer must avoid `Math.sin`-class functions differing across engines (V8 vs QuickJS) if client-prediction and server run different engines. Ship platform math (fixed-point or vetted impls)?
3. **How much API fits in the generation prompt** before quality drops — argues for a deliberately tiny (~50-call) surface, PICO-8-sized.
4. **Where Phaser-4-era training data lands by 2027** — revisit pinning.
5. **Moderation/abuse of generated multiplayer rooms** (names, drawings, griefing) — Verse's capability model suggests scoping what reducers can echo to other clients.
6. **Discord Activities as a distribution channel** (Playroom's wedge) — worth supporting even on a custom stack?
